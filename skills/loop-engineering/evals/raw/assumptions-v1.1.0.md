## Eval 14

适配路线：`agentic-loop`；当前模式：`Propose`。这项工作会反复发生，且问题筛选、修复和技术判断不能完全确定化。下面仅是非执行性提案，不授权访问 GitHub、修改仓库、启用调度或提交变更。

### Goal Contract

目标：持续把目标仓库中开放且标记为 `ai-fix` 的 bug issue，转化为以下两种可审阅结果之一：

- 经过独立验证的本地补丁与审阅包；
- 带证据的 `blocked` 记录，说明无法安全修复的原因和下一步。

验收标准：

- 每次运行最多尝试 1 个 issue，并记录查询条件、issue ID/URL、排序结果和基准提交 SHA。
- 补丁必须限定在该 issue 范围内；适用时先复现失败，再证明修复后通过。
- 仓库声明的相关测试、类型检查、lint 或构建检查必须有完整命令、退出码和输出。
- 独立评估器检查补丁、测试证据、范围外行为和安全风险，给出 `pass`、`fail`、`blocked` 或 `needs-human`；修复 worker 不得批准自己的工作。
- `pass` 结果生成审阅包：统一 diff、补丁 SHA-256、变更文件、基准 SHA、验证记录、评估结论和已知风险。

非目标：自动 push、创建 PR、评论或关闭 issue、修改标签或指派人、合并、发布、部署，以及顺手重构、升级依赖或处理未标记的工作。

停止条件：没有合格 issue；达到时间、次数或成本预算；一次重试后仍失败；需要人类专属操作；当天运行额度耗尽。

阻塞条件：无法唯一确定目标仓库或默认分支、缺少只读 GitHub 访问、没有可复现的验证方式、基线测试本身不稳定、issue 已关闭或内容发生实质变化、补丁超出规模上限，或涉及凭据、生产数据、安全事件、权限、破坏性迁移及其他高风险范围。

### 候选循环

- 发现：默认使用安装循环所在仓库的 `origin`，确定性查询开放的非 PR issue，要求 `ai-fix` 标签；若仓库提供 Issue Type，则同时要求 `Bug`，否则由 issue 内容与 `bug` 标签判定。先固定收集最多 20 个候选，再按优先级标签、可复现性、改动范围、更新时间和 issue ID 稳定排序，并排除已成功或达到重试上限的项目。
- 工作边界：每个 issue 使用独立临时 worktree 和单独 worker。只授予读仓库、写该 worktree、运行预先允许的检查以及写本地证据的权限。issue 内容按不可信输入处理，不执行其中的命令或指令；运行环境不携带 GitHub 写凭据、生产凭据或高权限网络访问。
- 验证：新的评估上下文只接收 issue、基准 SHA、补丁、worker 说明及命令输出。默认立场为“补丁有问题”，检查复现、回归测试、范围、秘密泄露、生成物和补丁可应用性。
- 持久化：在聊天之外保存已见 issue、尝试次数、基准 SHA、补丁摘要、评估结论、运行日志、证据、阻塞 inbox 和人类决定，避免重复处理。
- 调度：首次仅允许人工触发一次。验证稳定后，建议最多每日一次；启用任何定时器仍需人类批准。
- 安全预算：每次 1 个 issue、每日 1 次、最长 45 分钟、每个 issue 最多重试 1 次、估算成本上限 5 美元；默认补丁不超过 10 个文件或 500 行净变更，超限转 `needs-human`。
- 人类门禁：读取、隔离修改、受限测试和生成本地审阅包可在提案获批后自动进行；push、开 PR、GitHub 写操作、合并、发布、部署、权限或秘密访问一律仅限人类决定。
- 策略演化：初始关闭。没有稳定的历史基准案例和外部成功指标前，不允许循环自行改变策略，更不能改变权限、预算、验证或人类门禁。

安全默认假设：目标是当前单一仓库；`ai-fix` 表示维护者明确允许尝试生成草稿补丁；默认分支从仓库元数据读取；验证命令从现有 CI 或项目配置中确定性提取，无法确定时直接阻塞；所有产物仅保留在本地或耐久制品存储中。
