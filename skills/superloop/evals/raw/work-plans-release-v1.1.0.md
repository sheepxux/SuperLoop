# Eval 16

这适合设计成一个 **agentic loop**，当前选择 **Propose** 模式，执行形态是 **finite Work Plan**。它不是一次能可靠完成的单次任务，也不应把每个阶段做成不同 Skill：一个 Loop 负责完整目标，每个 Part 只是其中一个有界、可独立验收的工作单元。

**Goal Contract（待你确认）**

- 目标：从经确认的项目基础出发，按固定依赖关系完成各项交付物，逐 Part 独立验收，并由独立的最终评估者验证整合结果达到完整交付标准。
- 整体验收标准：
  1. 项目基础包明确记录最终交付物、目标用户、范围、非目标、约束、风险、权限边界和逐条可验证的验收标准；每项后续工作都能追溯到其中至少一条整体标准。
  2. 固定计划中的每个 Part 都生成其声明的持久化工件，并由非该 Part 工作者的评估者依据该 Part 的精确标准和证据给出 `passed`。
  3. 整合包包含全部已验收工件、接口与依赖检查结果、全量验证结果以及交付说明；不存在未解决的依赖冲突或缺失项。
  4. 最终 Goal Gate 在新上下文中逐条检查上述整体标准并通过；仅有全部 Part 通过不能代表项目完成。
- 非目标：运行中临时扩展范围；为不同 Part 新建 Skill 或嵌套 Loop；让工作者自我批准；未经单独的人类授权进行合并、部署、发布、付款、删除数据、外部通知或权限变更。
- 停止条件：最终 Goal Gate 通过；达到时间、费用或重试上限；你要求暂停；出现需要人类决定的高影响动作。
- 阻塞条件：最终交付物与验收人未明确、必需数据或工具不可用、验证方式未确定、依赖输入缺失，或所需权限尚未获批。

**固定 Work Plan**

默认 `maxPartsPerRun: 1`，严格逐 Part 推进；以下 ID、依赖、工件和门禁一旦获批就在运行期间冻结。

1. `foundation-baseline`（无依赖）
   - 工件：项目基础包。
   - Part Gate：目标、范围、非目标、利益相关方、约束、风险、权限边界和每条整体验收标准均已明确；所有未决歧义均被列为阻塞项或获得你的书面确认；独立评估者确认内容内部一致且可复现检查。
2. `delivery-blueprint`（依赖 `foundation-baseline`）
   - 工件：交付蓝图、依赖图、工件清单、验证矩阵。
   - Part Gate：依赖图无环且每条边有明确输入/输出；每个计划工件均映射到整体标准；每个 Part 都有明确证据、通过条件和责任边界；独立评估者确认没有未覆盖的目标或隐含依赖。
3. `core-capability`（依赖 `delivery-blueprint`）
   - 工件：可检查的核心交付包及其运行证据。
   - Part Gate：全部声明工件存在于本次运行的不可变快照中；基础包规定的核心行为检查全部通过；评估者确认没有越过范围、权限或非目标。
4. `dependent-capabilities`（依赖 `core-capability`）
   - 工件：依赖核心能力的其余交付包、接口证据和检查结果。
   - Part Gate：每项依赖输入均引用已通过的上游工件；该 Part 的精确功能、接口和异常路径检查全部通过；独立评估者确认预期工件完整。
5. `integration-hardening`（依赖 `core-capability` 和 `dependent-capabilities`）
   - 工件：整合候选、端到端验证报告、风险与回退说明。
   - Part Gate：整合、回归、边界与失败恢复检查全部通过；没有未解决的高严重度问题；工件摘要与实际字节的 SHA-256 一致。
6. `delivery-handoff`（依赖 `integration-hardening`）
   - 工件：最终交付候选、使用/运维说明、验收证据索引。
   - Part Gate：交付包可由未参与制作的人按说明复核；证据索引覆盖每项 Part 标准和整体标准；交付动作本身若涉及发布或外部影响，仍停在人类门禁前。

所有 Part 通过后才进入独立的 **Goal Gate**：评估者使用完整整合包和不弱于已批准验证矩阵的证据，逐条判断 Goal Contract 的全部验收标准。只有 Goal Gate 为 `passed`，Loop 才能标记 `completed`。

**候选运行边界**

- 每次只处理一个由运行时判定为 eligible 的 Part；一个隔离工作者产出工件，一个新上下文中的独立评估者给出门禁结论。
- 暂定每个 Part 最多 2 次纠正重试、单次最多 4 小时、单次费用上限 50 美元、全项目代理费用上限 300 美元；这些都是待确认的设计假设，不是已获授权的支出。
- 状态、运行记录、工件快照、证据、失败原因和人类决定都持久化在聊天之外。失败 Part 只能按原定义纠正，不会解锁其依赖项。
- 首次仅手动运行。批准候选方案不等于批准发布；基础包、任何不可逆动作以及最终交付各有独立的人类门禁。
- 如果要实质性增加、删除、重排或重释 Part，必须停止执行，创建带父提案哈希的新修订，重新取得绑定新摘要的人类决定，再编译、验证并通过受支持的状态迁移恢复；既有 Part 的通过结果不会自动转移。

当前 readiness 是 `needs-input`。请一次确认这三个会改变合同的问题：

1. 最终要交付的具体成果是什么，谁负责最终验收，必须满足哪些可观察标准？
2. 可使用哪些仓库、资料、工具和凭据，各 Part 应运行哪些可复现检查或由谁审阅？
3. 截止时间、费用上限、允许的动作，以及合并/部署/发布等高影响动作分别由谁批准？

收到这些答案后，下一步是把上述审阅包具体化为可验证的 `LoopProposal`，校验它并交给你按精确摘要确认；在你明确批准前，不会编译、脚手架化或执行。

# Eval 17

不能跳过当前失败的 Part 去做后面的 Part。固定 Work Plan 有两条不能绕过的约束：**依赖当前 Part 的后续 Part 仍然 locked**；而且 **Loop 在每个 Part Gate 以及独立的最终 Goal Gate 都通过之前都不能标记为 `completed`**。后续工作看起来更简单，不会改变它的依赖状态或当前失败结论。

下一步应当仍针对同一个 Part：

1. 保留本次失败工件、评估证据、检查退出码，以及该 Part 的精确定义摘要。
2. 根据评估者给出的有界失败原因修正同一个冻结定义，不降低标准、不换成更容易的任务，也不覆盖已留存的工件快照。
3. 在剩余重试预算内生成新的运行快照，并交给新的独立评估上下文，按同一组 Part 验收标准和有效验证命令重新判定。
4. 只有新的 `passed` 结果被持久化后，依赖它的 Part 才能解锁；全部 Part 通过后还要单独运行最终 Goal Gate。

如果失败源于缺少凭据、外部依赖或必须由人决定的事项，应记录为 `blocked` 或 `needs-human`，待条件确实解决后通过带操作者和原因的审计恢复操作重新打开；普通 resume 不会改变 Part 结论。若失败重试预算已经耗尽，则按既定失败策略停止并请求处理，不能用 reopen 或手工改状态伪造可继续。

只有当事实证明原计划本身存在实质性范围或依赖错误时，才可以停止当前执行并修订计划。那需要保留当前证据、创建绑定父提案摘要的新修订、让人重新批准精确的新摘要、重新编译和验证，并通过受支持的状态迁移恢复；不能在运行中临时重排，也不能假定旧的通过结果自动适用于改变后的 Part。

当前最小安全动作是把当前 Part 的评估结果、失败检查和剩余重试预算交给我，我会只为这个 Part 制定并执行下一次有界纠正。

# Eval 18

这适合 **agentic loop** 的 **continuous discovery** 形态，当前选择 **Propose** 模式。原因是“是否符合标准”和“如何形成简报”需要持续判断；它不是有限 Work Plan，也不应虚构一个最终完成日期。每天的一次运行有明确终点，但整个 Loop 会一直保持可暂停的持续状态。

**Goal Contract（待你确认）**

- 目标：每天从经批准的有限来源中收集自上次成功水位以来的新行业动态，依据冻结的入选标准筛选，并在存在合格新内容时生成一份可追溯、经独立检查的内部简报草稿。
- 验收标准：
  1. 每次运行保留来源清单、采集窗口、水位、候选数量和采集结果；覆盖范围可复核，失败来源不会被静默忽略。
  2. 每条候选内容具有稳定 ID、规范化链接、来源、发布时间和内容摘要哈希；已处理内容不会重复进入，除非出现可证明的实质更新。
  3. 每条入选内容逐项说明命中了哪条标准，并附可核验的原始来源证据；没有证据支持的事实不得写入简报。
  4. 简报遵循批准的模板，清楚区分事实、来源观点和分析判断，并由不同于生成工作者的评估者检查相关性、事实支持、去重、完整性和内部使用边界。
  5. 没有合格新内容时记录 `no-work` 运行回执，不生成空洞简报，也不把它当成任务“成功完成”。
- 非目标：监控整个互联网、擅自增加来源或修改入选标准、自动对外发布、把每日重复执行当成策略改进，或把没有固定终点的工作伪装成有限项目。
- 单次停止条件：达到候选/时间/费用上限，完成当日合格项的评估与简报，或遇到来源、权限、证据和人类门禁问题。
- Loop 停止或暂停条件：你要求暂停；连续预算耗尽；批准来源长期不可用；标准失效需重新审阅；合规或权限风险出现。

**候选运行方式**

- 暂定每天 09:00（Asia/Shanghai）运行，但第一次只做手动试跑；定时调度必须在精确提案摘要获你批准后才能启用。
- 先用确定性采集器读取最多 20 个批准来源，再按冻结规则去重和粗筛，最后才让模型做相关性判断。来源 allowlist 为空时直接阻塞，不允许开放式搜索代替审批。
- 暂定每次最多检查 100 条新候选、最多收录 10 条、运行 30 分钟、费用 5 美元、每个候选最多重试 2 次。达到上限后保留水位与待处理项，下一次继续，不静默丢弃。
- 工作单元是一条稳定 ID 的候选动态。隔离工作者为每条候选生成带证据的简报卡；独立评估者只依据冻结标准、来源快照和可检查证据判定。通过的卡片才可进入当日简报，完整简报再做一次独立质量检查。
- 在聊天之外持久化来源水位、seen IDs、内容哈希、筛选理由、证据快照、评估结论、运行预算、简报版本和人类决定。来源修订或正文更新产生新版本，不覆盖已接受证据。
- 默认只生成内部草稿文件，不自动发送或发布。向邮件、Slack、知识库或其他共享位置投递，以及新增来源、改变标准或提高预算，均需明确的人类授权。
- 暂不启用策略 evolution；只有以后存在稳定的历史样本、外部效果指标和可进行同案例对照的基准时，才另行提议。

当前 readiness 是 `needs-input`，因为批准来源和判定标准尚未给出。请一次确认：

1. 要监控哪个行业、地区和语言，什么条件算“符合标准”，哪些情况必须排除？
2. 允许使用哪些具体来源（最多 20 个），是否已有访问凭据、付费墙许可或合规限制？
3. 简报模板、存放位置和读者是谁；只生成草稿，还是允许自动投递到某个内部渠道，且每日时间/费用上限是多少？
