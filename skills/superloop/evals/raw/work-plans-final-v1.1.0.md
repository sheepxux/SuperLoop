# Eval 17

不能这样跳过。现有有限 Work Plan 一旦获批，失败的 Part 必须在同一个冻结定义和 `partDefinitionSha256` 下优先做有界修正与重试，不能把这次失败重定向到更简单的 Part，也不能在运行中删除、延后、重排或重新解释它。

这里有两条必须同时遵守的约束：

- 依赖该失败 Part 的后续 Parts 仍然是 locked，不能因为“比较简单”而解锁。
- 整个 Loop 仍不完整；只有每一个 Part Gate 都通过，并且随后独立的最终 Goal Gate 也通过，才可以标记为 `completed`。

下一步应依据独立评估器留下的失败证据，对当前 Part 做一个最小、明确的修正，写出新的运行专属产物快照，再由新的评估上下文按原 Part 的全部验收标准复验。旧的已接受快照不能覆盖，产出者也不能给自己最终通过结论。若重试预算已经耗尽，就按合同的失败策略停止，不能用 `resume` 或 `part reopen` 把耗尽重试的 `failed` 状态伪装成可继续。

如果失败说明的不是实现有误，而是已批准的 Part 定义、依赖或范围本身错了，那么也不能现场改计划：应先持久化失败证据并停止，基于原提案摘要创建新修订，明确修改 Parts/依赖/验收标准，重新取得人工的摘要绑定批准，再编译、验证并通过受支持的状态迁移恢复。定义发生变化时，旧 Part 即使 ID 相同，也不能自动继承原来的 `pass`。

# Eval 18

这个需求适合 **agentic loop / Propose 模式**，执行形态应是 **continuous discovery（持续发现）**，而不是有限 Work Plan：每天的新内容没有可预知的全集，Loop 也不应伪造一个固定完成时间。确定性采集、时间窗过滤和去重应先于模型判断；“是否符合标准”和“如何形成简报”才使用有边界的判断。

在生成可审阅提案前，需要你确认三个会改变边界的问题：

1. 要监控哪些明确来源、主题、地区和语言，允许使用哪些订阅、API 或凭据？
2. “符合标准”的逐条判定规则、最低证据要求和简报字段是什么；哪些来源可信，如何处理冲突、转载和不确定信息？
3. 简报只生成私有草稿，还是要写入或发送到某个内部系统；每天的时间/时区、候选数、成本上限、重试上限及发布批准人分别是什么？

可先审阅下面的非执行 Goal Contract 草案：

- **目标**：对每个批准的日度时间窗，从明确来源有界采集新内容，按获批标准识别符合项，并生成可追溯、可独立核验的内部简报草稿。
- **验收标准**：每次运行保留来源、查询窗口、采集回执、稳定内容 ID 和去重结果；每条入选内容逐条说明满足了哪些标准并附可访问来源；简报中的事实和结论能追溯到对应证据；独立评估器复核标准命中、遗漏、重复、引用和无依据表述。具体行业标准尚未提供，因此当前不能把提案标为可批准。
- **非目标**：无限制抓取网络、绕过付费或访问控制、自动改变筛选标准、自动对外发布、把无新内容当成任务成功、或把日常重复误称为策略改进。
- **单次停止条件**：完成一个日度窗口，或触及候选数、时间、成本、重试上限；无符合项时记录 `no-work` 和采集证据，不生成空洞简报，也不把它计作已验证成功。
- **Loop 级停止/暂停条件**：人工暂停或终止、来源条款改变、凭据失效、标准需要重审、预算用尽或独立证据不可用。它没有自然的 `completed` 状态。
- **阻塞条件**：来源清单或判定标准未定、必要权限/凭据缺失、目标位置不可写、发布需要的人工决定未取得，或无法进行独立核验。

候选 Loop 会以日度窗口 ID（例如绑定时区的日期）作为稳定工作单元：先按白名单来源确定性收集并去重，在每次运行的明确候选上限内排序和判断，产出一个耐久简报草稿及逐项证据，再交给与产出者不同的评估器。水位线、已见 ID、运行证据、简报快照、失败和人工决定全部持久化到聊天之外。溢出项进入有界 inbox，而不是扩大本次预算。

安全默认值是先手动试跑一次，确认漏报/误报和成本后再启用每天调度；在你明确授权前只生成草稿，不自动发送、发布或修改共享系统。可以暂以“每次一个日度批次、最多 50 个候选、最多 10 条入选、30 分钟、单项重试 1 次”为待确认的保守上限，任何超额内容留待下一次或人工处理；还需补充明确成本上限。策略演化保持关闭，直到有同一批历史案例、稳定评分标准和人工批准的推广流程。
