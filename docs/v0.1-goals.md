# v0.1.0 Goals

v0.1.0 should prove that Loop Engineering can be represented as a portable protocol instead of a vendor-specific prompt.

## Must Ship

- A valid `loop.yaml` schema.
- A persisted `state.json` schema.
- Evaluator and run-log schemas.
- `loopctl validate`.
- `loopctl init`.
- `loopctl render`.
- Thin adapters for Codex, Claude Code, ChatGPT, OpenClaw, and generic harnesses.
- Example loops for CI triage, dependency updates, and frontend QA.

## Must Encode and Validate

- Worker and evaluator roles must be distinct; the validator rejects identical roles.
- Every loop spec must declare persistence paths, and recorded runs must produce schema-valid state.
- A loop cannot schedule without budget and retry caps.
- A loop cannot request high-risk permissions without human-only gates.
- A loop cannot claim executable verification without commands.

## Deliberately Deferred

- A long-running cloud runner.
- Real GitHub API execution.
- Provider-specific subagent orchestration beyond rendered adapter files.
- Auto-merge or auto-deploy workflows.

## Deliberately Out of Scope for Protocol v1

These are boundaries, not omissions (rationale in [positioning.md](positioning.md)):

- Dependency graphs between items. Protocol v1 models a single-lane loop: bounded discovery, one item at a time. Deeply coupled task topologies are a different, harder problem.
- Process-level reward signals. Protocol v1 verification is one evaluator verdict per item; mid-run checkpoints are a possible later protocol extension.
- Any claim of model self-improvement. Loop-Engineering makes work accumulate across runs through persisted state; the reward stays external (tests, evidence, humans at gates).
